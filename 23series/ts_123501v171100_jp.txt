TS 23.501 v17.11.0
5G;
System architecture for the 5G System (5GS)
(3GPP TS 23.501 version 17.11.0 Release 17)


1 Scope
本書は、5Gシステムのステージ2システムアーキテクチャを定義する。
The present document defines the Stage 2 system architecture for the 5G System.
5Gシステムは、データ接続とサービスを提供する。
The 5G System provides data connectivity and services.

本仕様は、5GSとEPSの相互接続、5GS内の移動性、QoS、ポリシーコントロールと課金、認証など、ローミングおよび非ローミングのシナリオをすべての側面でカバーしている。SMS、位置情報サービス、緊急サービスなどの5Gシステム全体の機能も含まれる。
This specification covers both roaming and non-roaming scenarios in all aspects, including interworking between 5GS and EPS, mobility within 5GS, QoS, policy control and charging, authentication and in general 5G System wide features e.g. SMS, Location Services, Emergency Services.

ITU-T勧告I.130[11]は、電気通信サービスの特性付けのための3段階の方法を記述しており、ITU-T勧告Q.65[12]はその方法のステージ2を定義している。
ITU-T Recommendation I.130 [11] describes a three-stage method for characterisation of telecommunication services, and ITU-T Recommendation Q.65 [12] defines Stage 2 of the method.

TS 23.502[3]は、5Gシステムのステージ2手順とフローを含んでおり、本仕様の対応仕様である。
TS 23.502[3] contains the stage 2 procedures and flows for 5G System and it is a companion specification to this specification.

TS 23.503[45]は、5Gシステムのステージ2ポリシーコントロールと課金アーキテクチャを含んでおり、本仕様の対応仕様である。
TS 23.503 [45] contains the stage 2 Policy Control and Charging architecture for 5G System and it is a companion specification to this specification.

3 Definitions and abbrevitations
3.1 Definitions
本文の目的のために、TR 21.905[1]と以下に示す用語と定義が適用されます。本文で定義された用語は、TR 21.905[1]に同じ用語の定義がある場合でも、その定義よりも優先されます。
For the purposes of the present document, the terms and definitions given in TR 21.905 [1] and the following apply. A term defined in the present document takes precedence over the definition of the same term, if any, in TR 21.905 [1]. 

5G VN Group: 5G LAN型サービスのためにプライベート通信を使用するUEの集合です。
5G VN Group: A set of UEs using private communication for 5G LAN-type service. 

5G Access Network: 5Gコアネットワークに接続するNG-RANおよび/または非3GPP ANで構成されるアクセスネットワークです
5G Access Network: An access network comprising a NG-RAN and/or non-3GPP AN connecting to a 5G Core Network.

5G Access Stratum-based Time Distribution: TS38.331で規定された手順を用いて、無線インターフェースを介してUEに5GS時刻を提供するためにNG-RANが使用する時刻同期分配方法です。
5G Access Stratum-based Time Distribution: A time synchronization distribution method that is used by an NG-RAN to provide the 5GS time to the UE(s) over the radio interface using procedures specified in TS 38.331[28]. 

5G Core Network: 本文で規定されたコアネットワークです。5Gアクセスネットワークに接続します。
5G Core Network: The core network specified in the present document. It connects to a 5G Access Network.

5G LAN-Type Service: IPおよび/または非IP型通信を使用してプライベート通信を提供する5Gシステム上のサービスです。
5G LAN-Type Service: A service over the 5G system offering private communication using IP and/or non-IP type communications.

5G LAN-Virtual Network: 5G LAN型サービスをサポートできる5Gシステム上の仮想ネットワークです。
5G LAN-Virtual Network: A virtual network over the 5G system capable of supporting 5G LAN-type service.

5G NSWO：5G NSWOは、5GシステムとUEが提供する機能であり、5GSへの登録なしに5GSの認証情報を使用してWLANアクセスネットワークへの接続を可能にします。
5G NSWO: The 5G NSWO is the capability provided by 5G system and by UE to enable the connection to a WLAN
access network using 5GS credentials without registration to 5GS.

5G QoS Flow or QoS Flow: 5GシステムにおけるQoS転送処理の最も細かい粒度です。同じ5G QoSフローにマッピングされたすべてのトラフィックは同じ転送処理を受けます（例：スケジューリングポリシー、キュー管理ポリシー、レートシェーピングポリシー、RLC設定など）。異なるQoS転送処理を提供するには、別の5G QoSフローが必要です。
5G QoS Flow or QoS Flow: The finest granularity for QoS forwarding treatment in the 5G System. All traffic mapped to the same 5G QoS Flow receive the same forwarding treatment (e.g. scheduling policy, queue management policy, rate shaping policy, RLC configuration, etc.). Providing different QoS forwarding treatment requires separate 5G QoS Flow.

5G QoS Identifier: 5G QoSフローに提供される特定のQoS転送動作（例：パケットロス率、パケット遅延量）を参照するために使用されるスカラーです。これは、QoS転送処理を制御するノード固有のパラメータを参照する5QIによってアクセスネットワークで実装される場合があります（例：スケジューリングの重み、受入閾値、キュー管理閾値、リンク層プロトコル設定など）。
5G QoS Identifier: A scalar that is used as a reference to a specific QoS forwarding behaviour (e.g. packet loss rate, packet delay budget) to be provided to a 5G QoS Flow. This may be implemented in the access network by the 5QI referencing node specific parameters that control the QoS forwarding treatment (e.g. scheduling weights, admission thresholds, queue management thresholds, link layer protocol configuration, etc.).

5G System: 5Gアクセスネットワーク（AN）、5Gコアネットワーク、UEから構成される3GPPシステムです。
5G System: 3GPP system consisting of 5G Access Network (AN), 5G Core Network and UE.

5G-BRG：5G-BRGは、BBFで定義された5G-RGです。
5G-BRG: The 5G-BRG is a 5G-RG defined in BBF.

5G-CRG：5G-CRGは、DOCSIS MULPI で規定された5G-RGです。
5G-CRG: The 5G-CRG is a 5G-RG specified in DOCSIS MULPI [89]. 

5G-RG：5G-RGは、5Gコアに関してUEの役割を果たす5GCに接続できるRGです。セキュアエレメントをサポートし、5GCとN1シグナリングを交換します。5G-RGは、5G-BRGまたは5G-CRGのいずれかになります。
5G-RG: A 5G-RG is a RG capable of connecting to 5GC playing the role of a UE with regard to the 5G core. It supports secure element and exchanges N1 signalling with 5GC. The 5G-RG can be either a 5G-BRG or 5G-CRG.

Access Traffic Steering: アクセストラフィックステアリング：新しいデータフローのためにアクセスネットワークを選択し、このデータフローのトラフィックを選択されたアクセスネットワーク上で転送する手順です。アクセストラフィックステアリングは、3GPPアクセスと非3GPPアクセスの間で適用できます。
Access Traffic Steering: The procedure that selects an access network for a new data flow and transfers the traffic of this data flow over the selected access network. Access traffic steering is applicable between one 3GPP access and one non-3GPP access.

Access Traffic Switching: 進行中のデータフローの全てのトラフィックを、データフローの連続性を維持しながら別のアクセスネットワークに移動する手順。アクセストラフィックスイッチングは、3GPPアクセスと非3GPPアクセスの間で適用される。
Access Traffic Switching: The procedure that moves all traffic of an ongoing data flow from one access network to another access network in a way that maintains the continuity of the data flow. Access traffic switching is applicable between one 3GPP access and one non-3GPP access.

Access Traffic Splitting: データフローのトラフィックを複数のアクセスネットワークに分割する手順。トラフィックスプリッティングがデータフローに適用されると、データフローの一部のトラフィックは一方のアクセスを経由して、同じデータフローの別の一部のトラフィックは別のアクセスを経由して転送される。アクセストラフィックスプリッティングは、3GPPアクセスと非3GPPアクセスの間で適用される。
Access Traffic Splitting: The procedure that splits the traffic of a data flow across multiple access networks. When traffic splitting is applied to a data flow, some traffic of the data flow is transferred via one access and some other traffic of the same data flow is transferred via another access. Access traffic splitting is applicable between one 3GPP access and one non-3GPP access.

Allowed NSSAI: UEが現在の登録エリアでサービングPLMNで使用できるS-NSSAIの値を示す。
Allowed NSSAI: Indicating the S-NSSAIs values the UE could use in the Serving PLMN in the current Registration Area.

Allowed Area: UEが通信を開始できるエリアで、5.3.2.3節で規定されている。
Allowed Area: Area where the UE is allowed to initiate communication as specified in clause 5.3.2.3.

AMF Region: AMF Regionは、1つまたは複数のAMF Setで構成される。
AMF Region: An AMF Region consists of one or multiple AMF Sets. 

AMF Set: AMF Setは、特定のエリアとネットワークスライスをサービスするいくつかのAMFで構成される。AMF SetはAMF Region内で一意であり、同じネットワークスライスをサポートするAMFで構成される。AMF Regionごとに複数のAMF Setが定義される場合がある。同じAMF Set内のAMFインスタンスは地理的に分散しているが、同じコンテキストデータにアクセスできる。
AMF Set: An AMF Set consists of some AMFs that serve a given area and Network Slice(s). AMF Set is unique within an AMF Region and it comprises of AMFs that support the same Network Slice(s). Multiple AMF Sets may be defined per AMF Region. The AMF instances in the same AMF Set may be geographically distributed but have access to the same context data.

Application Identifier: 特定のアプリケーショントラフィック検出ルールにマッピングできる識別子。
Application Identifier: An identifier that can be mapped to a specific application traffic detection rule.

4 Architecture model and concepts
4.1 General concepts

5Gシステムのアーキテクチャは、データ接続とサービスをサポートするために定義されており、ネットワーク機能仮想化やソフトウェア定義ネットワーキングなどの技術を利用するデプロイメントを可能にします。
The 5G System architecture is defined to support data connectivity and services enabling deployments to use techniques such as e.g. Network Function Virtualization and Software Defined Networking.
5Gシステムのアーキテクチャは、特定された場所での制御面（CP）ネットワーク機能間のサービスベースの相互作用を活用するものとします。
The 5G System architecture shall leverage service-based interactions between Control Plane (CP) Network Functions where identified. 
いくつかの重要な原則と概念は以下のとおりです。
Some key principles and concept are to: 

	- ユーザプレーン（UP）機能と制御プレーン（CP）機能を分離し、独立したスケーラビリティ、進化、柔軟なデプロイメント（例えば、中央集権型や分散型（リモート）の場所）を可能にする。
	- Separate the User Plane (UP) functions from the Control Plane (CP) functions, allowing independent scalability,evolution and flexible deployments e.g. centralized location or distributed (remote) location.

	- 機能設計をモジュール化し、例えば、柔軟かつ効率的なネットワークスライシングを可能にする。
	- Modularize the function design, e.g. to enable flexible and efficient network slicing.

	- 可能な場合は、手順（すなわち、ネットワーク機能間の相互作用の集合）をサービスとして定義し、その再利用が可能になるようにする。
	- Wherever applicable, define procedures (i.e. the set of interactions between network functions) as services, so that their re-use is possible.

	- 各ネットワーク機能とそのネットワーク機能サービスが、必要に応じてサービスコミュニケーションプロキシを介して直接または間接的に他のネットワーク機能とそのネットワーク機能サービスと相互作用できるようにする。このアーキテクチャは、制御プレーンメッセージをルーティングするために別の中間機能（例えばDRAのようなもの）を使用することを妨げない。
	- Enable each Network Function and its Network Function Services to interact with other NF and its Network Function Services directly or indirectly via a Service Communication Proxy if required. The architecture does not preclude the use of another intermediate function to help route Control Plane messages (e.g. like a DRA). 
	
	- アクセスネットワーク（AN）とコアネットワーク（CN）との依存関係を最小限にする。このアーキテクチャは、異なるアクセスタイプ（例えば、3GPPアクセスや非3GPPアクセス）を統合する共通のAN - CNインターフェースを持つ、収束したコアネットワークで定義されている。
	- Minimize dependencies between the Access Network (AN) and the Core Network (CN). The architecture is defined with a converged core network with a common AN - CN interface which integrates different Access Types e.g. 3GPP access and non-3GPP access.

	- 統一された認証フレームワークをサポートする。
	- Support a unified authentication framework.

	- 「コンピュート」リソースと「ストレージ」リソースとを切り離した「ステートレス」なNFをサポートする。
	- Support "stateless" NFs, where the "compute" resource is decoupled from the "storage" resource.

	- ケイパビリティの露出をサポートする。
	- Support capability exposure.

	- ローカルサービスと中央集権型サービスへの同時アクセスをサポートする。低遅延サービスやデータネットワークへのローカルアクセスをサポートするために、UP機能はアクセスネットワークに近い場所にデプロイできる。
	- Support concurrent access to local and centralized services. To support low latency services and local access to data networks, UP functions can be deployed close to the Access Network. 

	- ホームルーテッドトラフィックと訪問先PLMNでのローカルブレイクアウトトラフィックの両方をサポートするローミングをサポートする。
	- Support roaming with both Home routed traffic as well as Local breakout traffic in the visited PLMN. 

4.2 Architecture reference model

4.2.1 General
この仕様書は、5Gシステムのアーキテクチャについて説明しています。5Gアーキテクチャはサービスベースとして定義され、ネットワーク機能間の相互作用は2つの方法で表現されます。
This specification describes the architecture for the 5G System. The 5G architecture is defined as service-based and the interaction between network functions is represented in two ways.

	- サービスベースの表現では、制御プレーン内のネットワーク機能（例：AMF）が、他の認可されたネットワーク機能に自身のサービスにアクセスすることを可能にします。この表現には、必要に応じて点対点の参照点も含まれます。
	- A service-based representation, where network functions (e.g. AMF) within the Control Plane enables other authorized network functions to access their services. This representation also includes point-to-point reference points where necessary. 
	- 点表現では、ネットワーク機能内のNFサービス間の相互作用を、任意の2つのネットワーク機能（例：AMFとSMF）間の点対点の参照点（例：N11）で表現します。サービスベースのインターフェースは、4.2.6節に記載されています。参照点は、4.2.7節に記載されています。 5GC制御プレーン内のネットワーク機能は、相互作用にサービスベースのインターフェースのみを使用しなければなりません（shall only use）。 注 1：このリリースの仕様では、1つのNF内のNFサービス間の相互作用は規定されていません。 NFとNFサービスは、直接通信と呼ばれる直接通信、またはSCP経由の間接通信と呼ばれる間接通信のいずれかで通信できます。通信オプションの詳細については、付録Eおよび6.3.1節と7.1.2節の条項を参照してください。 
	- A reference point representation, shows the interaction exist between the NF services in the network functions described by point-to-point reference point (e.g. N11) between any two network functions (e.g. AMF and SMF). Service-based interfaces are listed in clause 4.2.6. Reference points are listed in clause 4.2.7.

4.2.5 Data Storage architectures

図4.2.5-1に示されているように、5Gシステムのアーキテクチャでは、任意のNFがUDSFに対して非構造化データ（例えばUEコンテキスト）を格納・取得することができます。
As depicted in Figure 4.2.5-1, the 5G System architecture allows any NF to store and retrieve its unstructured data into/from a UDSF (e.g. UE contexts).

UDSFは、ネットワーク機能が所属するPLMNに属します。
The UDSF belongs to the same PLMN where the network function is located.

CP NFはそれぞれの非構造化データを格納するためにUDSFを共有することも、それぞれ独自のUDSFを持つこともできます（例えば、UDSFはそれぞれのNFに近い場所に配置されるかもしれません）。
CP NFs may share a UDSF for storing their respective unstructured data or may each have their own UDSF (e.g. a UDSF may be located close to the respective NF).

	注釈1: 本仕様書での構造化データとは、その構造が3GPP仕様書で定義されているデータを指します。
	NOTE 1: Structured data in this specification refers to data for which the structure is defined in 3GPP specifications.
	非構造化データとは、その構造が3GPP仕様書で定義されていないデータを指します。
	Unstructured data refers to data for which the structure is not defined in 3GPP specifications. 

	注釈2: 3GPPは、N18/Nudsfインターフェースを（参照によって可能ならば）規定する予定です。
	NOTE 2: 3GPP will specify (possibly by referencing) the N18/Nudsf interface.

As depicted in Figure 4.2.5-2, the 5G System architecture allows the UDM, PCF and NEF to store data in the UDR,including subscription data and policy data by UDM and PCF, structured data for exposure and application data (including Packet Flow Descriptions (PFDs) for application detection, AF request information for multiple UEs) by the NEF.

UDRは各PLMNに展開でき、以下のように異なる機能を提供できます。
UDR can be deployed in each PLMN and it can serve different functions as follows:

	- NEFがアクセスするUDRは、NEFが所在するPLMNと同じPLMNに属します。
	- UDR accessed by the NEF belongs to the same PLMN where the NEF is located.

	- UDMがスプリットアーキテクチャをサポートしている場合、UDMがアクセスするUDRは、UDMが所在するPLMNと同じPLMNに属します。
	- UDR accessed by the UDM belongs to the same PLMN where the UDM is located if UDM supports a split architecture.

	- PCFがアクセスするUDRは、PCFが所在するPLMNと同じPLMNに属します。 
	- UDR accessed by the PCF belongs to the same PLMN where the PCF is located.


	 注釈3：各PLMNに展開されたUDRは、ローミング加入者のアプリケーションデータを格納できます。
	 NOTE 3: The UDR deployed in each PLMN can store application data for roaming subscribers.

	 注釈4：ネットワークには複数のUDRが展開されており、それぞれが異なるデータセットやサブセット（例えば、加入データ、加入ポリシーデータ、露出用データ、アプリケーションデータなど）を収容したり、異なるNFのセットに対応したりできます。
	 NOTE 4: There can be multiple UDRs deployed in the network, each of which can accommodate different data sets or subsets, (e.g. subscription data, subscription policy data, data for exposure, application data) and/or serve different sets of NFs. 
	 UDRが単一のNFに対応し、そのデータを保存し、そのNFと統合できるような展開も可能です。
	 Deployments where a UDR serves a single NF and stores its data, and, thus, can be integrated with this NF, can be possible.

	 注釈5：図4.2.5-2に示されたUDRの内部構造は、情報提供のために示されているだけです。
	 The internal structure of the UDR in figure 4.2.5-2 is shown for information only.

Nudrインターフェースは、UDM、PCF、NEFなどのネットワーク機能（NFサービスコンシューマ）が、UDRに格納された特定のデータセットにアクセスし、UDR内の関連データの変更を通知するために読み取り、更新（追加、変更を含む）、削除、および購読するために定義されています。
The Nudr interface is defined for the network functions (i.e. NF Service Consumers), such as UDM, PCF and NEF, to access a particular set of the data stored and to read, update (including add, modify), delete, and subscribe to otification of relevant data changes in the UDR.

Nudrを介してUDRにアクセスする各NFサービスコンシューマは、変更する権限のあるデータのみを追加、変更、更新、または削除できるようにする必要があります。この権限は、UDRによって、データセットとNFサービスコンシューマの単位で、および可能であれば、UE、サブスクリプションの粒度ごとに行われます。
Each NF Service Consumer accessing the UDR, via Nudr, shall be able to add, modify, update or delete only the data it is authorised to change. This authorisation shall be performed by the UDR on a per data set and NF service consumer basis and potentially on a per UE, subscription granularity.

UDRのデータセットは、Nudrを介して各NFサービスコンシューマに公開され、保存される以下のデータが標準化されます。
The following data in the UDR sets exposed via Nudr to the respective NF service consumer and stored shall be standardized:

	- サブスクリプションデータ
	- Subscription Data,

	- ポリシーデータ
	- Policy Data,

	- 構造化されたエクスポージャ用データ
	- Structured Data for exposure,

	- アプリケーションデータ：アプリケーション検出と複数のUEに対するAF要求情報のためのパケットフロー記述（PFD）（5.6.7節で定義）
	- Application data: Packet Flow Descriptions (PFDs) for application detection and AF request information for multiple UEs, as defined in clause 5.6.7.

サービスベースのNudrインターフェースは、データセットによって公開される3GPPで定義された情報要素の内容と形式/エンコーディングを定義します。 
The service based Nudr interface defines the content and format/encoding of the 3GPP defined information elements exposed by the data sets.

また、NFサービスコンシューマは、UDRからオペレータ固有のデータセットにアクセスするだけでなく、各データセットに対するオペレータ固有のデータにもアクセスできるようにする必要があります。
In addition, it shall be possible to access operator specific data sets by the NF Service Consumers from the UDR as well as operator specific data for each data set.

	注釈6: オペレータ固有のデータとオペレータ固有のデータセットの内容と形式/エンコーディングは標準化の対象ではありません。
	NOTE 6: The content and format/encoding of operator specific data and operator specific data sets are not subject to standardization.

	注釈7: UDRに格納される異なるデータの構成は標準化されるべきではありません。
	NOTE 7: The organization of the different data stored in the UDR is not to be standardized.
